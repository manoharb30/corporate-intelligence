<objective>
## Goal

Fix four concrete correctness defects in `insider_cluster_service.detect_clusters` so that cluster detection respects the classification pipeline, matches the v1.0 mcap methodology, and is symmetric between buy and sell handling.

## Purpose

This is the enforcement half of Phase 17's earnings-proximity decision. Today the classification tag (GENUINE / FILTERED / NOT_GENUINE) is set by the pipeline but ignored by `detect_clusters`, which lets FILTERED and NOT_GENUINE transactions cluster anyway. That's the root defect behind:
- 17 of 58 current immature SP rows being backed by non-GENUINE underlying transactions (surfaced 2026-04-22)
- The AEVEX manual override failing to suppress the cluster
- The ~47 "fails-earnings" signals in the 141-cohort shipped in v1.1

Additionally, `detect_clusters` still uses a $10B midcap ceiling that contradicts `signal_performance_service.compute_conviction_tier`'s $5B ceiling (which was tightened on 2026-04-17 after p=0.018 analysis). And the buy branch doesn't exclude 10b5-1 planned purchases the way the sell branch does. And the mcap exception handler silently swallows yfinance failures.

All four defects live in the same file, all are low-risk, same-layer changes.

## Output

- Modified `backend/app/services/insider_cluster_service.py`
- Verified: next `compute_all` produces an immature cohort without FILTERED-backed clusters; 142 matured stay byte-identical (v1.2 invariant)
- SUMMARY.md documenting before/after cohort counts
</objective>

## AC-1: Classification filter enforces pipeline output

```gherkin
Given a set of InsiderTransaction rows in Neo4j where some are GENUINE,
      some FILTERED, some NOT_GENUINE, and some NULL-classification
When `detect_clusters(days=730, direction="buy")` is called
Then only transactions with `classification = 'GENUINE'` are considered
  for cluster aggregation; FILTERED, NOT_GENUINE, AMBIGUOUS, and NULL
  rows are excluded from the buyer count and total_value sum.
```

## AC-2: Midcap upper cap is $5B, matching signal_performance_service

```gherkin
Given `detect_clusters` classifies a cluster's conviction_tier
When the cluster's mcap is between $5,000,000,000 and $10,000,000,000
Then it is NOT classified as `strong_buy`; the upper sweet-spot boundary
  is $5B, identical to `signal_performance_service.compute_conviction_tier`.
```

## AC-3: 10b5-1 buy transactions excluded for symmetry with sell branch

```gherkin
Given the sell branch of `target_trades` filtering excludes
      `is_10b5_1 = true` transactions
When the buy branch runs under the same conditions
Then the buy branch also excludes `is_10b5_1 = true` transactions,
  producing symmetric handling for planned-trade flagging.
```

## AC-4: yfinance failures are logged, not silently swallowed

```gherkin
Given the exception handler at the `StockPriceService.get_market_cap` call
      inside `detect_clusters`
When an exception occurs (rate limit, missing ticker, API error, etc.)
Then a `logger.warning(...)` is emitted with the ticker symbol and the
  exception type/message; the fallback to value-only conviction tier
  still works as before.
```
